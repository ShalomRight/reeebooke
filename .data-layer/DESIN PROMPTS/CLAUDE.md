## What This System Is

A complete, layered prompt system for building, migrating, and changing
production-grade dashboards. It supports two stacks — **Next.js 16 (App Router)**
and **Laravel 12 + Inertia.js** — and covers everything from domain modeling
to pixel-level design tokens.

The system is made up of **14 files** in this folder.
You never load all of them at once.
You load only what the current goal requires.
`ORCHESTRATION.md` tells you exactly which files a given goal needs.

## Stack Decision — One Rule

```
Next.js 16 (App Router) when:
  - Consumer-facing, SEO matters
  - JavaScript / TypeScript-first team
  - Deploying to Vercel or edge
  - Custom UI is a competitive differentiator

Laravel 12 + Inertia.js when:
  - Ops-heavy, admin-heavy, internal tooling
  - PHP team or preference
  - Complex business logic, queues, background jobs
  - Multi-tenancy at scale
  - Rapid CRUD scaffolding more important than custom UI
```

For Laravel — Filament vs Inertia + React by zone:
```
Zone 1  Landing / Marketing    →  Blade + Tailwind  (or Inertia)
Zone 2  Client Portal          →  Inertia + React   (design prompts apply here)
Zone 3  Tenant Dashboard       →  Filament if ops/internal — Inertia if customers use it
Zone 4  Super Admin            →  Filament (almost always)
```

## Domain Complexity — Score Before Building

Before writing any code in any session involving data modeling:

```
Score one point per item that applies:

  [ ] Business rules are non-obvious
  [ ] Same concept means different things in different contexts
  [ ] Multiple things must react when one event occurs
  [ ] Status machine has 4+ states or non-linear transitions
  [ ] Money, availability, or scheduling logic is core
  [ ] Multi-developer team or long-lived system
  [ ] Multi-tenancy required
  [ ] Third-party integrations tightly coupled to domain logic

0–2  →  Standard CRUD. No domain layer.
3–4  →  Service layer only.
5+   →  Full DDD. The data layer prompt generates aggregates,
         value objects, domain events, repository interfaces,
         bounded context map, and application service stubs.
```

Score is recorded in `.claude/rules/domain-models.md`.

## Hard Rules — Never Break These

These apply in every session regardless of goal, stack, or phase.

**Code quality**
- No `any` types in TypeScript — use proper generics or `unknown` with type guards
- No raw `useState` for form state — React Hook Form + Zod / Inertia `useForm()` always
- No `<img>` tags in Next.js — `next/image` only
- No modifying existing migrations — always add new files
- No money stored as float — integer cents always (`amount Int // cents`)
- No hard deletes on user-facing data — soft delete with `deletedAt`

**Architecture**
- Schema before application code — migrations run before models, models before controllers
- Data layer prompt before master prompt — schema must exist before routing or forms
- Domain layer before HTTP layer — aggregates and services before controllers
- Controllers are thin — they translate HTTP to domain/service calls, nothing else
- Every query scoped to `organizationId` in multi-tenant apps — no exceptions

**Design**
- One depth strategy per design prompt — borders OR shadows, not both
- Semantic color by meaning — success = positive outcome, not just "green looks nice here"
- Never load all 13 prompt files at once — load only what the current goal requires
- Design prompts are zone-specific — a dp--*.md assigned to Zone 1 does not automatically apply to Zone 3

**Session discipline**
- Every non-obvious decision gets an entry in `.claude/build-log.md`
- Naming conventions in `.claude/rules/naming-conventions.md` are final — do not rename domain entities
- `.claude/AGENTS.md` is updated at the end of every session with current state

## Antigravity-Specific Notes

When running as an Antigravity agent:

**On initialization:**
1. Check if `.claude/AGENTS.md` exists in the project root
2. If yes — read it, identify current phase, proceed from next task
3. If no — read `ORCHESTRATION.md`, identify goal, open the matching orchestrator,
   run Phase 0 intake before any file generation or code writing

**On tool use:**
- File reads: use `read_file` on the specific prompt files the current goal requires
- File writes: all generated artifacts go to `.claude/` or `.data-layer/` — not to src/
- Web fetch: when a URL is provided as a design reference, fetch it before generating
  any design tokens — do not approximate what might be there
- When a screenshot is attached, run the full Phase 1 extraction from
  `tool--design-prompt-generator.md` before producing any token values

**On task sequencing:**
- Never generate application code before the schema is confirmed
- Never generate UI components before the design tokens are established
- Never skip the domain complexity score when the task involves data modeling
- Never begin a migration task without running the extraction prompt first

**On memory between runs:**
- There is no memory between runs — `.claude/AGENTS.md` is the memory
- At the end of every run, update `AGENTS.md` with:
  - Current phase
  - What was completed this run
  - What is next
  - Any decisions made that future runs need to know

**On parallel tasks:**
- The data layer prompt and the extraction prompt can run in parallel
- The theming prompt and the data layer prompt can run in parallel
- The master prompt, crud interactions prompt, and any secondary feature prompts
  must run after the data layer is confirmed — they depend on schema output
- Design prompt generation (tool) can run at any time — it is independent
