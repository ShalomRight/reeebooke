## File Roles — One Line Each

### Orchestrators

**`orchestrator--greenfield-builder.md`**
Runs Phases 0–7: intake → tech decision → data layer → design direction → route map →
feature classification → build order → rule file generation. Coordinates every other
file in sequence. You describe an idea; it sequences the system to produce a build plan.

**`orchestrator--migration.md`**
Runs the extraction prompt in read-only mode, audits the schema via the data layer
prompt, maps every source pattern to its target-stack equivalent, and produces a
phased migration plan. Three strategies: Big Bang, Strangler Fig, Frontend-Only Swap.

**`orchestrator--change.md`**
Four change categories (View Layer, API Extension, Feature Addition, Data Model Extension).
Boundary-maps the existing codebase before touching anything. Activates only the
specialist prompts relevant to what is actually changing. Contains the Fork Pattern
for large visual changes without touching the backend.

---

### Core Layer

**`dashboard-agent-master-prompt.md`**
The foundational rules that every build follows regardless of stack or goal.
Contains: Next.js 16 and Laravel 12 + Inertia stack tables, App Router structure,
Laravel route conventions, domain complexity scoring (0–8), DDD tactical patterns
(value objects, aggregates, domain events, repositories, application services),
folder structures for both stacks, CQRS, anti-corruption layer, outbox pattern,
circuit breaker, RBAC, multi-tenancy, accessibility, and deployment rules.
Also references Filament as a first-class Zone 3/4 option for Laravel.

---

### Parallel Layer

**`parallel-prompt--data-layer.md`**
Three modes: Complete (audit existing schema), Incomplete (fill gaps), Greenfield
(build from scratch). Produces: Prisma schema, Eloquent migrations + models,
TypeScript types, Zod schemas, API contracts (REST + Server Actions + Inertia),
bounded context map, event catalog, aggregate map, value objects, repository
interfaces, and application service stubs. The domain complexity score (from master
prompt) controls which of these outputs are generated.

**`parallel-prompt--extraction-migration.md`**
Reads an existing codebase and produces structured rule files. Five phases: ingestion,
rule file generation, migration translation, secondary feature prompt generation,
handoff package. The migration orchestrator runs this in read-only mode; the change
orchestrator runs it for boundary mapping. Contains the full Next.js ↔ Laravel
framework mapping tables.

---

### Secondary Layer

**`secondary-prompt--theming.md`**
Zone architecture (Landing / Portal / Dashboard / Admin), token system (:root → zone
→ component), warm dark mode rules, progressive disclosure as structure, semantic
color philosophy. Defines the slots that design prompts fill. Works with any
dp--*.md file. Per-org theme override pattern for multi-tenant apps.

**`secondary-prompt--crud-interactions.md`**
The complete interaction lifecycle: trigger → entry point → form → submission →
outcome → return → persistence. Defines: the three-tier trigger hierarchy,
ellipsis menu contract, form validation timing, server error field mapping,
four submission states, optimistic updates, toast contracts, navigation return map,
destructive action patterns, inline interactions, bulk actions.
Contains direct Laravel + Inertia equivalents for every Next.js pattern.
Ends with a quality-gate checklist (all items verified after build, not before).

---

### Design Prompts

Each `dp--*.md` is a complete, self-contained style engine. It plugs into the
theming prompt's zone architecture and provides exact token values. They are
interchangeable — swap one for another to completely change the visual character
of a zone without touching any other file.

**`dp--minimalist-monochrome`** — Stark editorial. Black/white/grey only. 0px radius everywhere.
Borders-only depth. 0–100ms binary motion. Bold factors: 8xl+ hero type, inverted
stats section, editorial pull-quotes, boxed drop cap.

**`dp--luxury-editorial`** — Warm cinematic. Alabaster, charcoal, gold accent. 0px radius.
Soft layered shadows. 500ms+ cinematic motion. Bold factors: visible vertical grid lines,
vertical image labels, asymmetric layout, grayscale→color image transition.

**`dp--terminal-cli`** — Phosphor green on near-black. JetBrains Mono everywhere. 0px radius.
1px borders only. 0–100ms linear. Bold factors: blinking cursor, shell prompt prefixes,
ASCII card borders, CRT scanline overlay.

**`dp--maximalism-dopamine`** — Five-color multi-accent on dark. Full radius on buttons,
3xl on cards. Glow + hard stacked shadows. Bouncy spring motion. Bold factors: floating
background shapes, massive background typography, pattern-on-pattern, systematic 5-color
rotation, animated gradient headlines.

---

### Tool

**`tool--design-prompt-generator.md`**
Input: screenshot, URL, Figma link, description, or any combination.
Output: a complete `dp--[name].md` file ready to plug into the theming system.
Runs the metacognitive scaffolding rule first (Purpose, Tone Commitment,
Differentiation, Anti-Convergence Check) before extracting any token.
Produces: § 0 Design Intent, § 1–6 token blocks, § 7 component rules, § 8 bold
factors, § 9 shadcn/ui implementation, § 10 anti-generic checklist, § 11 developer notes.

### Goal A — Build a new app from scratch

```
Entry:     orchestrator--greenfield-builder.md

The orchestrator activates these in sequence:
  Phase 2  → parallel-prompt--data-layer.md
  Phase 3  → secondary-prompt--theming.md
             + dp--[chosen style].md (one per zone)
  Phase 4  → dashboard-agent-master-prompt.md (conventions)
  Phase 5  → secondary-prompt--crud-interactions.md (for each entity)
             + any secondary feature prompts (calendar, canvas, etc.)

You do NOT manually activate these — the orchestrator does it.
Your only job: answer Phase 0 intake questions.
```

### Goal B — Migrate an existing app to a different stack

```
Entry:     orchestrator--migration.md

The orchestrator activates these in sequence:
  Phase 1  → parallel-prompt--extraction-migration.md (extraction mode)
  Phase 2  → parallel-prompt--data-layer.md (audit mode)
  Phase 3  → extraction-migration.md (migration map tables)
  Phase 4  → secondary-prompt--theming.md (if redesigning)
             + dp--[chosen style].md (if redesigning)

You do NOT manually activate these — the orchestrator does it.

```

### Goal C — Change an existing app (new feature / UI rebuild / API extension)

```
Entry:     orchestrator--change.md

The orchestrator selects specialist prompts based on change category:

  Category 1 (View Layer only):
    → secondary-prompt--theming.md
    → dp--[new or existing style].md
    → secondary-prompt--crud-interactions.md (for rebuilt forms)
    → dashboard-agent-master-prompt.md (Inertia conventions if Blade→Inertia)

  Category 2 (API Extension):
    → parallel-prompt--data-layer.md (API contracts section)
    → dashboard-agent-master-prompt.md (Route Handler / Controller conventions)

  Category 3 (Feature Addition):
    → parallel-prompt--data-layer.md (new entities only)
    → dashboard-agent-master-prompt.md (domain complexity score)
    → secondary-prompt--crud-interactions.md (new feature's interactions)
    → secondary feature prompt (if complex enough)

  Category 4 (Data Model Extension):
    → parallel-prompt--data-layer.md (extension mode — additive migration)
    → dashboard-agent-master-prompt.md (DDD patterns if score rises)

You do NOT manually activate these — the orchestrator does it.
Your only job: answer Phase 0 intake questions and identify the change category.
```

## How the Files Talk to Each Other

The communication between files happens through structured output files
written to `.claude/` and `.data-layer/`. Each prompt reads these files
at the start of its session rather than relying on conversational memory.

```
tool--design-prompt-generator
  └── writes → design-prompts/dp--[name].md

parallel-prompt--extraction-migration
  └── writes → .claude/rules/domain-models.md
  └── writes → .claude/rules/api-contracts.md
  └── writes → .claude/rules/naming-conventions.md
  └── writes → .claude/rules/feature-inventory.md

parallel-prompt--data-layer
  └── reads  ← .claude/rules/domain-models.md (if extraction ran first)
  └── writes → .data-layer/schema.prisma
  └── writes → .data-layer/types/
  └── writes → .data-layer/api-contracts/
  └── writes → .data-layer/domain/ (if score 5+)

secondary-prompt--theming
  └── reads  ← design-prompts/dp--[name].md
  └── writes → .claude/design/system.md

dashboard-agent-master-prompt
  └── reads  ← .data-layer/schema.prisma
  └── reads  ← .claude/rules/domain-models.md
  └── reads  ← .claude/design/system.md

secondary-prompt--crud-interactions
  └── reads  ← .data-layer/api-contracts/ (status machines, endpoints)
  └── reads  ← .claude/design/system.md (motion tokens, semantic colors)

orchestrators
  └── reads  ← all .claude/rules/ files
  └── writes → .claude/AGENTS.md (the session bootstrap file)
  └── writes → .claude/build-log.md
```

The `.claude/AGENTS.md` file is the most important output. Every new session
in a project starts by loading it. It contains the current build phase, the
next task, the critical rules, and pointers to the relevant specialist files.
Without it, every session starts blind.

## Session Bootstrap Protocol

Every session in an existing project should start the same way:

```
Step 1: Load AGENTS.md
  "Load .claude/AGENTS.md and tell me where we are."

Step 2: AGENTS.md points to the relevant files for this session
  If building a new entity → load master prompt + data layer output + crud interactions
  If changing the UI → load theming + the relevant dp--*.md
  If extending the API → load data layer + master prompt API conventions
  If migrating → load migration orchestrator + extraction output

Step 3: State the goal for this session in one sentence
  "I want to build the Booking entity CRUD today."
  "I want to change the dashboard from Blade to Inertia."
  "I want to generate a design prompt from this screenshot."

Step 4: The agent loads only the files relevant to that goal
  It does not load all 13 files. It loads what the goal requires.
```

## Filament Decision Matrix

For Laravel projects, this matrix determines which zones use Filament vs Inertia + React.
Referenced from `dashboard-agent-master-prompt.md`.

```
Zone 1 — Landing        → Always Blade + Tailwind or Inertia
Zone 2 — Client Portal  → Always Inertia + React (design prompts apply here)
Zone 3 — Tenant Dash    → Filament IF ops-facing / internal staff
                        → Inertia + React IF customers use it or design matters
Zone 4 — Super Admin    → Almost always Filament
```

If Zone 3 starts as Filament and later needs a custom design:
→ Use `orchestrator--change.md` (Category 1 — View Layer)
→ Fork pattern: keep Filament admin, build Inertia portal alongside
→ Same backend, same controllers — only the view layer forks

## DDD Activation Path

DDD is not on by default. It is activated by the domain complexity score.
The score lives in two places: `dashboard-agent-master-prompt.md` (§ Domain
Architecture Layer) and `parallel-prompt--data-layer.md` (§ Phase 1.5).

```
Score the project (0–8 points):
  + Business rules are non-obvious
  + Same concept means different things in different contexts
  + Multiple things react to a single event
  + Status machine has 4+ states or non-linear transitions
  + Money, availability, or scheduling logic is core
  + Multi-developer team or long-lived system
  + Multi-tenancy required
  + Third-party integrations tightly coupled to domain logic

0–2  → No domain layer. Standard controllers + services.
3–4  → Service layer. Extract logic into Service classes.
5+   → Full DDD. The data layer prompt generates:
         .data-layer/domain/bounded-context-map.md
         .data-layer/domain/event-catalog.md
         .data-layer/domain/aggregate-map.md
         .data-layer/domain/value-objects/
         .data-layer/domain/repositories/
         .data-layer/domain/events/
         .data-layer/domain/services/ (stubs)
```

## Common Mistakes to Avoid

**Loading all 13 files at once.**
The system is designed to load only what the current goal needs.
Loading everything creates noise and conflicting instructions.
Let the orchestrator tell you which files are active for this session.

**Starting a session without AGENTS.md.**
If AGENTS.md doesn't exist yet, run the appropriate orchestrator first.
It is the session bootstrap — without it, the agent has no project context.

**Running the data layer prompt after the master prompt.**
The data layer prompt feeds the master prompt — schema must exist before
routing, forms, or interactions are built. Run data layer first, always.

**Generating a design prompt and never registering it.**
After `tool--design-prompt-generator` produces a new `dp--*.md`, it must be:
  1. Saved in `/design-prompts/`
  2. Assigned to a zone in `.claude/design/system.md`
  3. Loaded into `secondary-prompt--theming.md`'s context
  Otherwise it exists but has no effect on any build.

**Using the migration orchestrator to add features.**
Migration moves what exists. New features during a migration are recorded
as post-migration tasks and handled by `orchestrator--change.md` afterward.
Mixing the two is the most common cause of migration failures.

**Skipping the domain complexity score.**
The score is not optional. A score of 0 and a score of 6 produce completely
different output from `parallel-prompt--data-layer.md`. Running without
scoring defaults to the wrong output set.
